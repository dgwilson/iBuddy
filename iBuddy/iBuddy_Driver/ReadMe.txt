The device reports to be a HID device.
- interface 0 is reporting UsagePage as Generic Desktop / Usage Page as Keyboard/Keypad
- interface 1 is reporting UsagePage as Generic Desktop / Usage Page LED

So this Kext aims to "Mask" or block out interface 0 from the Operating System and leave Interface 1 available as a HID device so we can connect to it.


-----

copy iBuddy_Driver.kext to /System/Library/Extensions

sudo chown -R root:wheel iBuddy_Driver.kext/
sudo kextutil -mtv iBuddy_Driver.kext/

Diagnostics - now we really have to get stuck into things:
We need to know what's going on under the hood. To investigate this, find the Console application - it's in the /Application/Utilities folder. Look for entries related to USB Missile Launcher NZ. Then start reading in date/time sequence.
If you see an entry that looks like this "USBMissileControl: FindInterfaces: kIOReturnExclusiveAccess (some other task has the device opened already) - Unable to open interface" then we've got a challenge. Essentially this means that the operating system got it's hands on the launcher before we did and this is no good to us. Actually this is the reason the KEXT file exists - it's a kernel extension usually for printer drivers or other low level devices - only in our case the KEXT has no actual program code it just tells the operating system via a plist to ignore the devices. "/System/Library/Extensions/USB Missile Launcher All Drivers.kext".
